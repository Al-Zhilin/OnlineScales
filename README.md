## *Электронные весы с удалённым мониторингом на ESP32-S3*

### *Краткий обзор:*

- ESP32-S3 N16R8 в качестве основного МК с light sleep для экономии заряда батареи
- Взвешивание через тензодатчики с 24-битным АЦП HX711
- Температурный датчик DS18B20 и адаптивная компенсация температурного дрейфа тензодатчиков через рекуррентный МНК с автоматическим выбором лучшей полиномиальной модели
- Передача данных через GSM-модем SIM800L; данные принимает VK, Callback API проксирует их в Yandex Cloud Function
- Хранение данных в Yandex Database с визуализацией в Yandex DataLens
- Оповещение по email при аномалиях через Yandex Cloud Function
- Поддержка работы через WiFi (в разработке)
- Адаптивные модели поведения при неисправностях датчиков и модема
- Многорежимная индикация на встроенном WS2812 RGB-светодиоде

### *Принцип работы:*

Прошивка реализована в виде конечного автомата (FSM). Устройство большую часть времени проводит в режиме light sleep (по умолчанию 30 секунд), периодически просыпаясь для измерений и отправки данных раз в 90 минут (при нормальной работе и 30 минут в режиме калибровки, данные по умолчанию).

**Цикл работы:**

1. **`WAKEUP_SENSORS`** — пробуждение HX711 и ожидание его инициализации
2. **`MEASURE`** — снятие показаний весов и температуры, обновление EMA-фильтров, диагностика датчиков
3. **`START_MODEM`** — инициализация SIM800L, установка GSM-соединения (через PPP)
4. **`DATA_SEND`** — отправка HTTPS-запроса к VK API с данными о весе, температуре и напряжении батареи + системная информация при надобности
5. **`SLEEP_MODEM`** — корректное выключение модема
6. **`ERROR_HANDLING`** — анализ накопленных ошибок, логика их устранения
7. **`SLEEP_SENSORS`** — перевод HX711 в сон, настройка источников пробуждения GPIO/таймер, уход в light sleep

Между 90-минутными (default значение) циклами отправки устройство остаётся в измерительной петле (`WAKEUP → MEASURE → SLEEP`) без включения модема.

**Пример сообщения в VK:**

```
Отчет от весов:
Текущий вес: 12.34 кг
Температура: 23.5 °C
Напряжение батареи: 4.12В
[SC:1 E:00 RB:3]
```

> [!NOTE]
> Мета-тег `[SC:N E:HH M:N/N RB:N]` в конце сообщения содержит диагностическую информацию (при наличии):
> - `SC` — причина последнего выключения (`esp_reset_reason()`)
> - `E` — маска накопленных ошибок в hex (см. [Поведение при ошибках](#поведение-при-ошибках))
> - `M` — состояние FSM и шаг модема на момент краша (только при аварийном перезапуске)
> - `RB` — оставшийся бюджет перезагрузок

### *Алгоритм компенсации температурного дрейфа:*

Тензодатчики подвержены температурному дрейфу: при изменении температуры показания смещаются даже без изменения нагрузки. Для компенсации реализован класс `AdaptiveRLS` — адаптивный рекуррентный МНК (RLS) с автовыбором модели. Дрейф нуля решается пользователем: перед использованием рекомендуется нагрузить весы на некоторое время рабочим/чуть большим весом в выключенном состоянии.

**Как работает:**

1. Переключите тумблер `CALIB_SWITCH_PIN` в режим калибровки. На весах должен находиться известный эталонный груз.
2. Прошивка параллельно обучает три регрессионные модели: линейную, квадратичную и кубическую — на парах (вес, температура), поступающих каждые 30 секунд.
3. Каждые 30 минут в VK приходит диагностическое сообщение с прогрессом 
калибровки: количество точек, диапазон температур, текущая неопределённость модели.
4. Контролируйте показатель "неуверенности" в чате ВК/на графике Yandex Data Lens: низкий показатель в течении длительного времени и стабильная близость к эталонному весу (т.е. изначальной массе груза не весах. **НЕ "текущий вес"!. Эта строчка в сообщениях и график на дашборде - показатель эталонной массы + влияние дрейфа!**) - повод завершить калибровку. **Важно:** калибровка должна "пройти" весь температурных диапазон будущего использования весов - иначе коменсация в экстраполирующем диапазоне будет некорректной.
5. Переключите тумблер обратно для завершения. Прошивка автоматически выбирает наилучшую модель (по средней абсолютной ошибке leave-one-out с штрафами за сложность).
6. Параметры модели сохраняются в LittleFS (`/calib.dat`) и применяются при каждом измерении и после перезагрузки.

> [!NOTE]
> Если при переключении тумблера в начальное положение «Калибровка» данных накоплено недостаточно (мало точек или слишком малый диапазон температур) — модель не будет сохранена, что отобразится в светодиодной индикации и сообщении в VK.

### *Световая индикация:*

Индикация реализована на встроенном WS2812 и управляется тумблером `LED_SWITCH_PIN` (при выключенном тумблере светодиод не работает для экономии энергии в автономном режиме). Очередь из 5 отчётов в программе позволяет сохранять результаты нескольких последовательных событий. Например, в конце каждог полного цикла (измерения датчиков + работа модема) происходит индикация как статуса работы датчиков, так и модема, последовательно, разделяя события паузой. См. расшифроку кодов индикации в следующем разделе.

**Дыхание (пока происходит длительный процесс):**

| Цвет | Событие |
|------|---------|
| Синий | Инициализация и настройка SIM800L |
| Оранжевый | Подключение к сети (PPP/GPRS) |
| Пурпурный | HTTPS-запрос к VK API |
| Жёлтый | WiFi: режим точки доступа *(в разработке)* |
| Зелёный | WiFi: подключение к сети *(в разработке)* |

**Вспышки (короткий отчёт по результату события):**

| Цвет | Количество | Событие |
|------|-----------|---------|
| Зелёный | 1 | Датчики: всё ОК |
| Красный | 1 | Датчики: ошибка DS18B20 или HX711 |
| Зелёный | 2 | Модем: успешная отправка данных |
| Красный | 2 | Модем: аппаратная ошибка (SIM800L не ответил) |
| Красный | 3 | Модем: ошибка сети (PPP не поднялся) |
| Красный | 4 | Модем: ошибка сервера (VK API или HTTPS) |
| Голубой | 3 (быстро) | Тарирование выполнено успешно |
| Оранжевый | 3 (быстро) | Тарирование: ошибка (датчик не готов или высокий шум) |
| Белый | 2 | Калибровка: начата |
| Белый | 1 (длинная) | Калибровка: модель сохранена |
| Красный | 4 (быстро) | Калибровка: ошибка (мало точек или малый диапазон температур) |
| Синий | 1 (длинная) | Данные модели выведены в Serial |

### *Управление:*

**Кнопка** (пин `BUTT_PIN`). Подключается на **INPUT_PULLUP**:

| Ввод | Действие |
|------|----------|
| Удержание | Тарирование (обнуление весов) |
| 1 нажатие + удержание | Принудительная внеплановая отправка данных |
| 2 нажатия + удержание | Вывод параметров сохранённой модели калибровки в Serial |
| 3 нажатия + удержание | Сброс данных калибровки из LittleFS |

**Тумблер режима** (пин `CALIB_SWITCH_PIN`):

| Положение | Действие |
|-----------|----------|
| Выкл | Штатный режим измерений |
| Вкл | Режим калибровки температурного дрейфа |

**Тумблер индикации** (пин `LED_SWITCH_PIN`):

| Положение | Действие |
|-----------|----------|
| Вкл | Светодиодная индикация активна |
| Выкл | Светодиод отключён |

> [!NOTE]
> Тарирование отклоняется при повторном вызове в течение 5 секунд (защита от удержания кнопки). Новый оффсет сохраняется в LittleFS (`/offset.dat`) только после успешной проверки тишины сигнала.

### *Поведение при ошибках:*

При любой ошибке устанавливается соответствующий бит в маске `rtc_error_mask`, которая хранится в RTC-памяти и переживает light sleep и перезагрузки. Маска включается в мета-тег следующего сообщения.

**Коды ошибок (биты маски, hex):**

| Бит | Значение |
|-----|---------|
| 0 (`0x01`) | Ошибка DS18B20 |
| 1 (`0x02`) | Ошибка HX711 |
| 2 (`0x04`) | Аппаратная ошибка модема |
| 3 (`0x08`) | Ошибка сети (PPP timeout) |
| 4 (`0x10`) | Принудительная перезагрузка |
| 5 (`0x20`) | Ошибка сервера (VK API / HTTPS) |

**Бюджет перезагрузок (`reboot_budget = 3`):**

- Ошибки связи — включается `is_retry_mode`: повторные попытки каждые 5 минут вместо 90.
- Каждые 3 цикла с ошибками или при зависании датчиков тратится одна «жизнь» и выполняется `ESP.restart()`.
- После исчерпания бюджета устройство продолжает работу по стандартному таймеру без перезагрузок.
- При успешном цикле без ошибок счётчики и маска сбрасываются, бюджет восстанавливается до 3.

#### *Коэффициент калибровки весов:*

В `src/main.cpp` при создании объекта `ScalesManager` первым аргументом передаётся `calibration_factor` — коэффициент перевода сырых единиц HX711 в граммы:

```cpp
ScalesManager scales(6.7, DT_PIN, SCL_PIN);  // 6.7 — calibration_factor, подбирается вручную
```

Для подбора коэффициента включите соответствующий режим в `Config.h`:

```cpp
#define WEIGHT_KOEFF_SELECTION 1  // 1 — режим подбора коэффициента, 0 — выкл
```

constexpr const char* APN = "internet";  // замените на APN вашего оператора при необходимости

### *Облачная часть:*

*— раздел в разработке —*

Устройство отправляет данные в VK через API (`messages.send`). VK Callback API получает событие и перенаправляет его в Yandex Cloud Function. Функция:

- Сохраняет данные в Yandex Database (YDB)
- Отправляет оповещения по email при аномальных значениях
- Данные из YDB используются Yandex DataLens для построения графиков KPI
